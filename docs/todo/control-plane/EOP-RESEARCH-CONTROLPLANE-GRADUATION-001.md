### A2 — `OB_POC_CONTROL_PLANE_ENFORCE_VERBS` exact semantics

CONFIRMED, read directly (`rust/src/agent/control_plane_envelope_store.rs:31-44`):
```rust
pub(crate) fn from_env() -> Self {
    let raw = std::env::var("OB_POC_CONTROL_PLANE_ENFORCE_VERBS").unwrap_or_default();
    Self(raw.split(',').map(|s| s.trim().to_string()).filter(|s| !s.is_empty()).collect())
}
pub(crate) fn is_enforced(&self, verb_fqn: &str) -> bool { self.0.contains(verb_fqn) }
```
- **Matching rule**: exact string equality against the full verb FQN (e.g. `cbu.confirm`), via a `HashSet<String>::contains`. No prefix, glob, or domain-level matching — `cbu.*` would not match anything; each verb must be listed individually, comma-separated, trimmed of whitespace.
- **Default**: unset or empty env var → empty set → `is_enforced()` always `false` for every verb → `AdmissionDecision::NotEnforced` (confirmed by the module's own doc comment at line ~25: "deliberately never populated by this tranche's own code").
- **Granularity**: **enforcement is per-verb-FQN, and that check is path-agnostic** — `is_enforced()` is called from inside `admit_in_scope`/`execute_verb_admitting_envelope` (`ObPocVerbExecutor`), which is the single production implementation both Path A (`step_executor_bridge.rs:553`) and Path D (`bus_runtime.rs`) call into today. There is no path-scoping parameter anywhere in `EnforcedVerbs`/`is_enforced`. **Consequence: you cannot enforce `cbu.confirm` on Path A only while leaving it shadow on Path D as currently built** — if `cbu.confirm` is in the enforced set, every call to `execute_verb_admitting_envelope` for that FQN from any caller (A or D today; B/C once they reach the same admitting entry point) is enforced simultaneously. **Smallest enforceable unit today: one verb FQN, globally across every path that reaches `ObPocVerbExecutor::admit_in_scope`.** Per-path graduation (as the runbook's §3 "graduation order" implies — Path A, then B, then C, then D) is currently an artifact of *which paths reach the admitting entry point at all* (A and D do; B and C structurally cannot yet, per T9.3's ledger entry — see Q-block B for the wrinkle there), not a first-class per-path enforcement dimension in the mechanism itself. If Path B/C are wired to reach the SAME `ObPocVerbExecutor::admit_in_scope` (Option (iii) in Q-block B3), then the "graduate Path A only" framing becomes impossible to express with today's `EnforcedVerbs` alone — a verb enforced for Path A is automatically enforced for B/C/D too, once they share the admitting call. This is a genuine design fork worth surfacing to the plan: **either accept global-per-verb enforcement (simpler, but a single verb's graduation implicitly graduates every path that dispatches it) or add a path dimension to `EnforcedVerbs`** (more precise, more state to reason about). Not decided here — OPTIONS, flagged as directly relevant to how the graduation plan sequences Path A vs B/C/D.
